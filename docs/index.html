A comprehensive Blender 5.0+ addon for seam creation, modular rigging, skinning, animation generation, and LLM bridge integration. You model; BlenderTools handles everything after.

  2. Rig it — Use Viewport Overlay to click-place modules, load a config preset, or import FBX > Scan Skeleton > Apply Wrap Rig

◢ Seams

Six UV seam algorithms plus preset combinations. All work on the active mesh object.

Methods

Seams by Angle

Marks edges as seams where the dihedral angle between adjacent faces exceeds a threshold.

When to use
General-purpose, good first pass for any model.
ParameterRangeDescription
Angle Threshold0–180° (default 30°)Lower = more seams
Clear ExistingboolRemove existing seams first

Seams by Material

Marks edges where adjacent faces have different material assignments.

When to use
Models with distinct material regions (skin vs clothing, metal vs rubber).

Seams by Hard Edge

Marks seams at sharp/hard edges using Blender 5.0's sharp_edge attribute.

When to use
Hard-surface models where you've already marked sharp edges for shading.

Island-Aware Seams

Iterative algorithm that starts with angle-based seams and adds more at high-distortion edges.

When to use
When you need control over UV island count and stretch.
ParameterRangeDescription
Max Islands0 = unlimitedCap on island count
Max Stretch0–1Target distortion threshold

Seams by Projection

Marks seams based on projection mapping directions.

When to use
Simple shapes that map well to box/cylinder/sphere projections.
ModeStrategy
BOXSeams where face normals change dominant axis
CYLINDERSeams at cap transitions + one vertical cut
SPHERESingle longitude cut

Seam Presets

Apply a predefined combination of seam methods.

PresetMethods
character_bodyAngle (60°) + Material + Hard Edge
hard_surfaceHard Edge + Angle (35°) + Material
organicIsland-Aware + Material

Clear All Seams

Removes all seams from the active mesh. Useful before applying a new seam method from scratch.

Bridge API

python blender_api.py seam --object Cube --preset hard_surface

⚙ Rigging

Modular rigging system. Each body part is a self-contained module that creates its own bones, constraints, and controls. Modules are assembled into a complete rig. For imported skeletons, the Skeleton Scanner wraps existing bones with controls.

Universal Three-Tier Architecture

CTRL → MCH → DEF
All 13 rig modules and the wrap rig use the same three-tier bone architecture:

DEF bones (deform): Only carry COPY_TRANSFORMS from their paired MCH bone. Never carry rig logic. use_deform=True.
MCH bones (mechanism): Carry all rig logic (FK copy, IK, DAMPED_TRACK, STRETCH_TO, etc.). Parented to their CTRLs. use_deform=False.
CTRL bones (controls): User-facing control bones for animation. use_deform=False.

This means rig logic never touches your deform bones directly. DEF bones always follow MCH via a clean COPY_TRANSFORMS constraint.

Name Bones Overlay
Click “Name Bones” in the Skeleton Scanner panel to activate an interactive overlay. Hover over any bone to see a clickable circle (white = unlabeled, green = BT-named). Click to open a dialog with Type, Side, and Role dropdowns that renames the bone automatically. The scanner auto-detects BT-named bones at 100% confidence before trying name maps or heuristics.

Four Workflows

Viewport Module Overlay (Interactive Placement)

  1. Select an armature and open BlenderTools > Rigging
  2. Click "Toggle Module Overlay" — white circles appear on bone heads and tails (alpha 0.3)
  3. Hover over a circle — it brightens to alpha 0.9 with a thick white outline
  4. Click a circle to open the module picker menu
  5. Choose a module type — it is placed at that bone position
  6. For modules with bone slots (piston, wheel): two menu entries appear — "Add" (creates new DEF bones) and "Map Existing Bones" (enters assignment mode)
  7. Repeat for all body parts, then "Generate Rig"
Overlay visibility
The overlay is automatically hidden in Pose mode. It only shows in Object and Edit mode.

Click-to-Place

  1. Select an armature
  2. Position the 3D cursor where you want the module
  3. Click "Add Rig Module"
  4. Choose module type, name, and side
  5. Repeat for all body parts
  6. Click "Generate Rig"

Config File

  1. Select an armature
  2. Click "Load Rig Config"
  3. Choose a preset (biped_human, quadruped, mech_walker)
  4. Click "Generate Rig"

Skeleton Scanner (Wrap Existing Bones)

For imported skeletons (Mixamo, UE Mannequin, or any source) that have deform bones but no control rig:

  1. Select the armature
  2. Open Rigging > Skeleton Scanner
  3. (Optional) Click “Name Bones” to label bones with the BT convention via interactive overlay
  4. Click "Scan Skeleton" — auto-detects bone roles via BT convention (Step 0), name maps, then position heuristics
  5. Review chains and bone assignments — edit module types, roles, or skip bones. Configure FK/IK/Snap per chain.
  6. Click "Apply Wrap Rig" — three-layer rig (CTRL → MCH → DEF) wraps the original bones with FK/IK controls
Non-destructive

Wrap Rig Constraint Flow

Wrap Rig Three-Tier Details
The wrap rig follows the universal three-tier pattern with FK/IK switching on the MCH layer:
DEF (original bones) → COPY_TRANSFORMS from MCH (always influence=1.0, never toggled)
MCH → COPY_TRANSFORMS from CTRL-FK (LOCAL space, toggled for FK) + IK constraint (toggled for IK)
CTRL → FK controls that animators manipulate, plus IK target/pole bones

The MCH→CTRL-FK constraint uses LOCAL space so that child chains (e.g. fingers parented to hand MCH) correctly inherit world-space movement from their MCH parent when the parent chain is in IK mode.

Cross-chain hierarchy: Chains are processed in dependency order (root → spine → neck/head → arm/leg → finger → generic). The first CTRL bone in each chain auto-parents to the nearest ancestor’s CTRL from a parent chain. Moving spine controls moves all attached limbs.

IK bone sizes: IK target and pole bones are sized proportionally to their chain (30% of the lower bone’s length), not tiny hardcoded values.

FK/IK Build Config vs. Runtime State

Build configuration and runtime state are separate:

  • fk_enabled / ik_enabled — Control whether FK/IK controls are generated during build (build-time settings).
  • ik_snap — Per-chain toggle (build-time). When enabled, IK chain_count is clamped to 2 for stable FK/IK switching with pose snapping. Default ON for arms/legs, OFF for others.
  • ik_active — Tracks which mode is currently active at runtime. All chains start in FK mode after applying the wrap rig.

FK/IK Toggle with Snapping

FK/IK switching works on any chain that has IK constraints, not just arm/leg. The bt.toggle_fk_ik operator handles the switching.

Pose Snapping
When ik_snap is enabled on a chain (2-bone IK like arms/legs), toggling preserves the current pose:
IK → FK: FK bones snap to match the IK-solved pose
FK → IK: IK target snaps to chain end, pole angle recalibrated

Panel modes change based on wrap rig state:

  • Before wrap rig exists (config mode): FK/IK/Snap config checkboxes visible. Bone mapping, batch skip, and unmapped bones sections all visible.
  • After wrap rig exists (rig mode): FK/IK toggle appears when any bone in an IK-capable chain is selected (DEF, MCH, CTRL-FK, IK target, or pole). Bone mapping, batch skip, and unmapped bones sections are hidden.

IK control visibility: IK target and pole bones are placed in per-chain bone collections (IK_arm_L, IK_leg_R, etc.) that start hidden. Toggling to IK shows the collection for that chain; toggling to FK hides it. Only active IK controls are visible in the viewport.

Pole Angle Calibration

IK pole angles are automatically calibrated during rig assembly using a depsgraph-based approach. The system sets pole_angle=0, lets Blender’s IK solver run via dependency graph evaluation, measures the resulting bend twist, and applies the correction. This works universally for any bone roll or orientation.

Bone Slot Mapping

Modules with bone slots (piston, wheel) support mapping to existing bones rather than creating new DEF bones. This is accessed via the viewport overlay’s “Map Existing Bones” menu entry.

Assignment Mode Workflow

  1. Click "Map Existing Bones" in the overlay menu
  2. A header text appears in the viewport showing which slot to fill (e.g. “Click bone for: Cylinder”)
  3. Click bones to assign them sequentially to each slot — assigned bones turn green
  4. RMB to skip a slot, Enter to confirm early, ESC to cancel

What Happens When Mapping

  • Module creates MCH bones at mapped bone positions (parented to their CTRLs)
  • Module creates CTRL bones (parented to the mapped bone’s skeleton parent)
  • Module skips creating DEF bones — uses the existing bones instead
  • Existing bones get COPY_TRANSFORMS from MCH (clean, non-destructive)

Piston Architecture

The piston uses a fixed, deterministic three-tier architecture. Default constraint is DAMPED_TRACK (rotation only, rigid). Configurable stretch_mode: "none" (default, DAMPED_TRACK), "cylinder", "rod", or "both" (uses STRETCH_TO for selected parts).

CTRL-Base (inherits cylinder’s skeleton parent)
   +-- MCH-Cylinder  --DAMPED_TRACK-->  CTRL-Target
CTRL-Target (inherits rod_parent’s skeleton parent)
   +-- MCH-Rod       --DAMPED_TRACK-->  CTRL-Base

DEF-Cylinder  <-- COPY_TRANSFORMS --  MCH-Cylinder
DEF-Rod       <-- COPY_TRANSFORMS --  MCH-Rod

Bone slots: Cylinder (outer body), Rod (inner shaft), Rod_Parent (explicit parent for CTRL-Target).

Generic piston usage
Works for any scenario: same-parent (decorative pistons), cross-joint (shock absorbers spanning knee/elbow), or free-standing. MCH bones are always parented to CTRLs; CTRLs inherit the mapped bone’s skeleton parent for automatic movement.

Wheel Architecture

CTRL-Position (inherits axle’s skeleton parent)
   +-- CTRL-Rotation
   |      +-- MCH-Wheel  (COPY_ROTATION from CTRL-Rotation)
   +-- MCH-Axle           (COPY_LOCATION from CTRL-Position)

DEF-Axle   <-- COPY_TRANSFORMS --  MCH-Axle
DEF-Wheel  <-- COPY_TRANSFORMS --  MCH-Wheel

Floor Contact

Prevents IK feet from going below a configurable floor level, with optional auto toe bend on ground contact.

  • Floor constraint: LIMIT_LOCATION (min Z, world space) on each leg IK target
  • Toe auto-bend: TRANSFORM constraint on the toe bone — maps IK target Z to toe X rotation. Active only in IK mode (toggled by FK/IK switch).
  • Panel: Toggle button in Skeleton Scanner panel when wrap rig has legs. Configure floor level, toe bend on/off, max angle.

Root Motion Extraction

One-click workflow using the reference-object pinning method (Pierrick Picaut). All key controllers (IK hands/feet + COG) are pinned to baked reference empties. The source bone (COG/hips) additionally has its XY travel + Z rotation extracted to the root bone.

Panel is under Animation > Root Motion. Operators: bt.rm_setup, bt.rm_finalize, bt.rm_cancel.

Root Motion Extraction

One-click workflow using the reference-object pinning method (Pierrick Picaut). Pins key controllers to baked empties, freeing the root bone for motion extraction.

  • Pinned bones — All controllers that hold position when root moves (IK hands/feet + COG/torso). Each gets a baked reference empty + COPY_TRANSFORMS constraint back.
  • Source bone — One of the pinned bones (COG/hips/torso) whose XY travel + Z rotation are additionally extracted to the root bone. Defines the character’s travel path.
  • Root bone — Receives the extracted locomotion data. Created at origin if missing.
  1. Auto-detect source, root, and pinned controllers (from wrap rig data or heuristics)
  2. Setup — creates reference empties, pins controllers, extracts root travel
  3. Polish root curves in Graph Editor (optional)
  4. Finalize — bakes all controllers with visual keying, removes empties and constraints

Panel: Animation > Root Motion. Config phase shows bone selection; active phase shows Finalize/Cancel buttons.

Thread Safety

Important
All Blender operations are dispatched to the main thread via bpy.app.timers.register(). The HTTP thread blocks until completion (30s timeout).

Response Format

// Success
{"success": true, "seams_marked": 42, ...}

// Error
{"success": false, "error": "Object not found: Cube"}

Scene Summary Format

The scene summary is designed to be token-efficient for LLM consumption:

  • Object name, type, location
  • Mesh: vertex/face count, materials, vertex groups, shape keys
  • Armature: bone count, bone names, module count
  • Camera: lens, clip start/end
  • Light: type, energy
  • Parent relationships and active action names

⤴ Export & Scale

Scale rigs mid-project and export UE-ready FBX files with correct unit scaling.

The Problem

Blender’s FBX exporter Apply Unit Scale option inconsistently scales geometry vs. animation translation keyframes when importing into Unreal Engine. This causes animations to look correct in Blender but play at wrong scale in UE.

The Solution
Scale the rig 100× in Blender, then export FBX at 0.01 global scale with apply_unit_scale=False. BlenderTools automates this entire process.

Scale Rig

Rescale a rig by any factor without breaking animations or constraints.

What gets scaled:

  • Armature object — scale applied via transform_apply
  • Location keyframes — all values + bezier handles (via channelbag API)
  • Constraint distances — LIMIT_DISTANCE, STRETCH_TO, FLOOR, LIMIT_LOCATION
  • Child meshes — scaled and applied
  • Rig config — position arrays and distance-related options

UI Usage

  1. Select the armature
  2. BlenderTools > Export > Scale Rig > "Scale Rig"
  3. Enter a factor (e.g. 2.0 to double, 0.5 to halve)

Bake to DEF

When using a wrap rig (CTRL → MCH → DEF), animations are driven through constraints. Before exporting, use "Bake to DEF" to bake the constraint-driven motion onto the DEF bones as keyframes. This ensures clean FBX export since only DEF bones (use_deform=True) are included in the exported file.

python blender_api.py rig-bake-to-def --armature Armature

Export to UE

One-click export that duplicates the hierarchy, scales 100×, exports FBX at 0.01, and cleans up.

Bone mapping in configs
The bone_mapping option maps module slots to existing skeleton bones. When set, the module skips creating DEF bones and wraps the existing ones with MCH/CTRL controls instead. Use the viewport overlay’s "Map Existing Bones" for interactive assignment.
